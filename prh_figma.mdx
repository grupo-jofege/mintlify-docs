Guia prático para transformar o mapa de telas do Portal de Requisição de RH em uma estrutura de trabalho no Figma, com foco em layout, biblioteca de componentes, estados e tokens.

## Diretriz de plataforma

O produto deve nascer como `site web`, com experiência otimizada para desktop e boa adaptação para larguras menores dentro do contexto de navegador. Ao mesmo tempo, a linguagem visual e a arquitetura de componentes devem ser desenhadas para permitir evolução futura para `app`, sem recomeçar o design system.

Na prática, isso significa:

- priorizar padrões de navegação e densidade adequados para web
- evitar soluções muito dependentes de hover como única forma de interação
- criar componentes com variantes que possam ser reutilizadas em contextos touch
- separar bem `layout responsivo`, `componente` e `regra de negócio`
- pensar desde cedo em tokens e componentes com comportamento cross-platform

## Princípios de desenho web-first, app-ready

### Web-first

Prioridades do MVP:

- leitura rápida em desktop
- tabelas, filtros e formulários com boa densidade de informação
- navegação clara em ambiente web
- ações principais sempre visíveis

### App-ready

Cuidados desde já:

- áreas de clique confortáveis para touch
- componentes que funcionem sem hover
- headers, cards e formulários que possam empilhar bem em telas menores
- evitar layouts que dependam exclusivamente de tabelas largas
- modelar navegação e ações em blocos reaproveitáveis

## Página 01. Fundamentos

Essa página concentra as decisões visuais base.

### Breakpoints de referência

Como o produto nasce para web, mas pode virar app no futuro, vale trabalhar com alguns frames e breakpoints de referência desde o começo:

- `Desktop / 1440`
- `Desktop / 1280`
- `Tablet / 1024`
- `Mobile referência / 390 ou 428`

Mesmo que o MVP não tenha tela mobile fechada, manter uma referência menor ajuda a validar se a arquitetura visual está pronta para evoluir.

### Tokens que vale desenhar primeiro

#### Cores

Estruture por grupos:

- `Brand`
- `Neutral`
- `Success`
- `Warning`
- `Danger`
- `Info`

Sugestão de aplicação:

- `Brand`: CTAs, links principais, foco
- `Neutral`: superfícies, bordas, textos
- `Success`: aprovado, concluído, sucesso operacional
- `Warning`: atenção, pendência, carregamento parcial
- `Danger`: erro, reprovação, falha
- `Info`: mensagens contextuais e estados informativos

#### Tipografia

Crie estilos separados para:

- `Display`
- `Heading`
- `Title`
- `Body`
- `Caption`
- `Label`

Hierarquia mínima recomendada:

- `Heading/L`
- `Heading/M`
- `Title/L`
- `Title/M`
- `Body/M`
- `Body/S`
- `Label/M`
- `Label/S`

#### Espaçamento

Adote uma escala consistente. Exemplo:

- `4`
- `8`
- `12`
- `16`
- `24`
- `32`
- `40`
- `48`
- `64`

Ao escolher espaçamentos, prefira uma escala que funcione tanto em web quanto em contextos mobile futuros.

#### Raios

Pelo menos:

- `4`
- `8`
- `12`
- `16`
- `24`

#### Sombra

Separe em poucos níveis:

- `Shadow/Sm`
- `Shadow/Md`
- `Shadow/Lg`

#### Bordas

Padronize:

- `Border/Subtle`
- `Border/Default`
- `Border/Strong`
- `Border/Interactive`

## Layout por tela

## Login

### Estrutura

- área externa com fundo neutro ou institucional
- card central com largura controlada
- spacing vertical generoso

### Observação futura para app

O bloco central deve conseguir virar uma composição vertical simples, sem depender de margens laterais muito grandes.

### Hierarquia

1. Marca
2. Título
3. Descrição curta
4. Campos
5. CTA
6. Feedback

## Minhas Solicitações

### Estrutura

- header com CTA à direita
- faixa de filtros abaixo do header
- tabela ocupando o corpo principal

### Observação futura para app

A listagem deve ter versão possível em cards, porque tabelas largas não escalam bem para uma experiência mobile.

### Hierarquia

1. Título
2. Nova solicitação
3. Filtros
4. Lista
5. Paginação

## Detalhe da solicitação

### Estrutura

- header contextual
- grid de cards resumo
- blocos de dados
- timeline
- anexos
- aprovação

### Sugestão de grid

- 12 colunas
- cards resumo em 3 ou 4 colunas
- blocos principais em largura maior
- sidebar opcional para ações, se quiserem destacar aprovação

### Observação futura para app

Evite depender de sidebar fixa na regra principal da tela. Se quiserem usar sidebar no web, tratem como variação de layout, não como única composição possível.

## Formulário

### Estrutura

- header fixo ou estável
- formulário em largura confortável
- divisões por seção
- rodapé fixo

### Observação futura para app

As seções do formulário devem empilhar naturalmente e funcionar bem em fluxo vertical único.

### Hierarquia

1. Contexto organizacional
2. Dados da solicitação
3. Justificativa
4. Ações

## Administração

### Estrutura

- header com ação de adicionar
- tabela de configurações
- modal ou drawer de edição

### Observação futura para app

Essa é a tela menos prioritária para adaptação mobile no curto prazo, então pode nascer mais orientada a desktop. Ainda assim, os formulários e blocos internos devem continuar reutilizáveis.

## Convenção de nomenclatura no Figma

Para a biblioteca ficar limpa, sugiro:

### Componentes

`Componente / Variante / Estado`

Exemplos:

- `Button / Primary / Default`
- `Button / Primary / Loading`
- `StatusBadge / Approved / Default`
- `TextField / Default / Error`
- `PageHeader / WithDescription / WithCTA`

### Frames

`Tela / Cenário`

Exemplos:

- `Login / InvalidCredentials`
- `Solicitações / Empty`
- `Detalhe / ApprovalEnabled`
- `Formulário / SubmitError`

## Observações de produto para refletir no visual

- O fluxo mais importante do MVP é abrir e acompanhar `Aumento de Quadro`
- A listagem e o detalhe precisam destacar `status` com muita clareza
- O formulário precisa parecer confiável, guiado e fácil de preencher
- A aprovação deve transmitir segurança, rastreabilidade e responsabilidade
- A área administrativa pode ter visual mais denso, mas não deve perder legibilidade
- O produto deve parecer nativamente confortável no web desde o primeiro dia
- A base visual deve permitir transição futura para app sem reconstrução completa da biblioteca
