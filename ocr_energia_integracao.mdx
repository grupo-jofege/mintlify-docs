## V360

### Autenticação OAuth2

Endpoint:

```text
POST https://jofege.virtual360.io/bus/oauth/token
```

Content-Type:

```text
application/x-www-form-urlencoded
```

Body:

```text
grant_type=client_credentials
client_id={client_id}
client_secret={client_secret}
```

Uso no fluxo:

- o node `Auth V360` obtém o token
- o token é enviado em `Authorization: Bearer {token}`

### Consulta incremental de documentos

Endpoint:

```text
GET https://jofege.virtual360.io/nf/api/v2/tax_documents
```

Parâmetros usados:

```text
limit={limite}
extra_fields=document_xml_url,document_pdf_url
q[id_gt]={ultimo_id_v360}
q[model_type_eq]=electricity
query_template=completed
```

### Consulta pontual para renovar o PDF

Antes do download, o workflow consulta novamente o documento específico para obter uma URL assinada válida.

Regra importante:

- `document_pdf_url` não deve ser persistida no banco
- a URL deve ser consumida logo após a consulta

## Recognition

### Envio do PDF

Endpoint:

```text
POST https://app.recognition-ia.com/recognition-api/api/v1/services/batches
```

Headers:

```http
RECOGNITION-SERVICE-API-KEY: {api_key}
```

Body `multipart/form-data`:

- `model = ENERGIA`
- `files = <arquivo PDF>`

### Polling do lote

Endpoint:

```text
GET https://app.recognition-ia.com/recognition-api/api/v1/services/batches/{documentId}
```

Status tratados como pendentes:

- `CREATED`
- `STARTING`
- `PROCESSING`

Status finais:

- `PROCESSED`
- `CANCELED`
- `TIMEOUT`

### Estratégia de polling

- intervalo de `5 segundos`
- até `120 tentativas`
- timeout lógico de `10 minutos`

## API de e-mail

### Objetivo

Ao final da execução, o orquestrador chama o subworkflow de resumo, que monta um HTML e envia os indicadores operacionais.

Endpoint:

```text
POST {config.urlJofegeMail}
```

Payload esperado:

```json
{
  "from": "integracao.fluig@jofege.com.br",
  "subject": "Contas de Energia V360 - Sucesso 10 | Erro 1 | Pendente 2",
  "body": {
    "contentType": "HTML",
    "content": "<!DOCTYPE html>..."
  },
  "recipients": ["lucas.dutra@jofege.com.br"]
}
```

## Mapeamentos principais

### V360 -\> CONTA\_ENERGIA\_DOCUMENTO

| Origem | Destino |
| --- | --- |
| `id` | `ID_V360_DOCUMENTO` |
| `model_type` | `MODEL_TYPE` |
| `customer_identification_number` | `CUSTOMER_IDENTIFICATION_NUMBER` |
| `supplier_identification_number` | `SUPPLIER_IDENTIFICATION_NUMBER` |
| `number` | `DOCUMENT_NUMBER` |
| `series` | `DOCUMENT_SERIES` |
| `issue_date` | `ISSUE_DATE` |
| `total_value` | `TOTAL_VALUE` |

### Recognition -\> CONTA\_ENERGIA\_CONSUMO

O retorno do OCR é tratado e transformado em um conjunto padronizado de itens de consumo antes da inserção em banco.

Campos típicos:

- `ordemConsumo`
- `empresa`
- `prestador`
- `numInstalacao`
- `consumoDescricao`
- `consumoFinal`
- `consumoTarifa`
- `consumoValor`
- `dataDocumento`
- `dataVenc`
- `notaFiscal`

## Contratos internos entre workflows

### Orquestrador -\> Download Arquivo

Campos principais:

- `idV360Documento`
- `modelType`
- `documentNumber`
- `issueDate`
- `totalValue`
- `config`

### Download Arquivo -\> Recognition

Campos principais:

- `idV360Documento`
- `documentNumber`
- `recognitionDocumentIdBanco`
- `config`

### Orquestrador -\> Envio Email

Campos principais:

- `config.urlJofegeMail`
- `config.destinatarios`
- `config.executionStartedAt`

## Observações importantes

- o `documentId` do polling é o mesmo retornado no envio ao `Recognition`
- o fluxo permite retomada de itens já enviados
- erros de envio, timeout e falhas de persistência são centralizados em `CONTA_ENERGIA_DOCUMENTO`
