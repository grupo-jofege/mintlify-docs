## 2. Pauta e Discussão

### 2.1 Contexto do Problema

- O **Portal Meu RH da TOTVS** suporta apenas gestores CLT vinculados a uma **chapa** no RM. Gestores externos ou PJs ficam fora desse fluxo.
- Como consequência, solicitações como **aumento de quadro, promoção, transferência, demissão** não podem ser abertas por esses gestores no portal nativo.
- A solução é construir um **portal próprio** integrado ao TOTVS via API, que ofereça essa funcionalidade de forma centralizada.

### 2.2 Escopo do Sistema

O projeto foi dividido em **duas fases**:

- **Fase 1 (MVP — 1 mês):** Foco exclusivo em **Aumento de Quadro** (solicitação de contratação de novos colaboradores), incluindo formulário, acompanhamento de status e aprovação.
- **Fase 2 (Evolução):** Inclusão de outros tipos de requisição (aumento de salário, promoção, transferência, demissão, férias) e implementação do **Motor de Markdown** para customização dinâmica de telas.

### 2.3 Fluxos de Aprovação

Foram identificados **dois fluxos distintos**:

1. **Fluxo Gestor Direto:** O próprio gestor cria a solicitação → aprovação segue direto para o RH.
2. **Fluxo Delegado:** Um assistente/funcionário cria a solicitação em nome de um gestor (ex.: Ronailson em nome do Alexandre Pio) → a solicitação passa por uma **etapa adicional de aprovação do gestor** antes de chegar ao RH.

Ambos os fluxos são processados via **"Minhas Aprovações"** do TOTVS (sistema de workflow), gerenciado pelo time de Sistemas.

### 2.4 Autenticação

- Não será possível reaproveitar o login do Fluig diretamente, pois a API do RM exige autenticação própria com **usuário e senha do TOTVS**.
- Será necessária uma **tela de login dedicada**, com credenciais do RM.
- A gestão de acessos no RM ficará sob responsabilidade do time de **Sistemas**.

### 2.5 Motor de Markdown (Fase 2)

- Igor propôs um mecanismo onde um **template em Markdown** seria armazenado em banco NoSQL (ex.: MongoDB) e processado dinamicamente pelo portal.
- Isso permitiria ao time de Sistemas customizar os campos exibidos na tela de aprovação **sem necessidade de alteração de código**.
- O grupo discutiu alternativas (HTML, banco relacional) e concluiu que o Markdown é adequado pela simplicidade de edição e pela ampla disponibilidade de bibliotecas de renderização.
- **Willian** sugeriu criar uma tela de cadastro de templates para centralizar validações.

### 2.6 Arquitetura de APIs

- Serão **duas APIs separadas**:
  - **API ACL (Anti-Corruption Layer):** camada de integração com o TOTVS RM (já parcialmente desenvolvida por Igor).
  - **API do Portal de Requisições de RH:** lógica de negócio própria do portal.
- A separação foi justificada pela reusabilidade: a camada de tickets/aprovações será compartilhada com o projeto de Solicitações de Obras.

### 2.7 Banco de Dados

- Dados de solicitações: **TOTVS RM** como fonte principal.
- Templates de Markdown (Fase 2): banco **NoSQL** (MongoDB ou Redis), para evitar estrutura relacional complexa na serialização de templates.

## 6. Mapa de Telas e Rotas (Fase 1)

### 6.1 Visão Geral das Rotas

```text
/login
/portal
  ├── /solicitacoes              → Lista de solicitações do usuário
  │     └── /:id                → Detalhe + aprovação de uma solicitação
  └── /nova-solicitacao

/admin (restrito a Administradores)
  └── /parametros               → Configuração de RH Central e permissões
```

---

### 6.2 Detalhamento por Tela

---

#### 🔐 Tela: Login

**Rota:** `/login`

**Objetivo:** Autenticar o usuário diretamente no TOTVS RM (credenciais próprias do RM, sem SSO com AD ou Fluig).

**Elementos:**

- Campo de **usuário** (login RM)
- Campo de **senha**
- Botão "Entrar"
- Mensagem de erro em caso de credenciais inválidas

**Comportamento:**

- Após autenticação bem-sucedida, redireciona para `/portal/solicitacoes`
- Token de sessão do RM armazenado em memória (sem localStorage)
- Usuários sem permissão no portal recebem mensagem de acesso negado

---

#### 📋 Tela: Minhas Solicitações (Listagem)

**Rota:** `/portal/solicitacoes`

**Objetivo:** Exibir todas as solicitações criadas pelo usuário logado (ou em seu nome), com status atualizado do TOTVS.

**Elementos:**

- Tabela/lista com colunas: **Nº da Solicitação**, **Tipo**, **Requisitante Real**, **Data de Criação**, **Status** (badge colorido)
- **Filtros:** por status (Em andamento / Aprovado / Cancelado / Concluído), por período, por empresa/filial
- **Botão "Nova Solicitação"** → redireciona para `/portal/nova-solicitacao`
- Paginação

**Comportamento:**

- Status consultado em tempo real via API ACL → "Minhas Aprovações" do RM
- Solicitações criadas _em nome_ de outros também são exibidas, com indicação do criador e do requisitante real

---

#### 🔍 Tela: Detalhe da Solicitação \+ Aprovação

**Rota:** `/portal/solicitacoes/:id`

**Objetivo:** Exibir todos os dados de uma solicitação específica, o histórico de movimentações e — quando aplicável ao perfil do usuário — permitir **aprovar ou reprovar**.

**Elementos:**

- **Cabeçalho:** Nº da solicitação (portal e RM), tipo, data de criação, status atual
- **Dados do formulário:** empresa, filial, seção, função, requisitante real, quantidade de vagas, obra/projeto, previsão de início, justificativa
- **Histórico de movimentações:** etapa, responsável atual, data, comentários
- **Anexos** vinculados ao atendimento no RM
- **Seção de Aprovação** _(visível apenas para aprovadores na alçada)_:
  - Botão "Aprovar"
  - Botão "Reprovar" (com campo de justificativa obrigatório)

**Comportamento:**

- Dados carregados via API ACL (endpoint de detalhe do atendimento)
- A seção de aprovação é exibida condicionalmente com base no perfil do usuário e na etapa atual do fluxo
- Aprovação/reprovação avança a etapa no "Minhas Aprovações" via API

> 💡 **Nota de modularização:** Esta tela deve ser desenvolvida como **módulo reutilizável**, pois o mesmo padrão de aprovação será utilizado em projetos futuros (ex.: Solicitações de Obras).

---

#### ➕ Tela: Nova Solicitação (Seleção de Tipo)

**Rota:** `/portal/nova-solicitacao`

**Objetivo:** Ponto de entrada para criação de uma nova solicitação. Na Fase 1, direciona diretamente para o formulário de Aumento de Quadro.

**Elementos:**

- Card ou botão "**Aumento de Quadro**" → redireciona para `/portal/nova-solicitacao/aumento-de-quadro`
- _(Fase 2: outros tipos de requisição aparecerão aqui como novos cards)_

---

#### 📝 Tela: Formulário de Aumento de Quadro

**Rota:** `/portal/nova-solicitacao/aumento-de-quadro`

**Objetivo:** Capturar todos os dados necessários para abrir uma solicitação de Aumento de Quadro no TOTVS RM.

**Campos do formulário:**

| Campo | Tipo | Obrigatório |
| :-- | :-- | :-- |
| **Empresa / Coligada** | Select (lista do RM) | ✅ |
| **Filial** | Select (filtrado por empresa) | ✅ |
| **Seção / Departamento** | Select (filtrado por filial) | ✅ |
| **Função** | Select (lista de funções do RM) | ✅ |
| **Requisitante Real** | Select/busca de usuários | ✅ |
| **Obra / Projeto** | Select ou texto | ✅ (quando aplicável) |
| **Quantidade de Vagas** | Número inteiro positivo | ✅ |
| **Previsão de Início** | Date picker (≥ hoje) | ✅ |
| **Justificativa** | Textarea | ✅ |
| **Observações** | Textarea | ❌ |

**Comportamento:**

- Validação em tempo real dos campos obrigatórios antes do envio
- Ao submeter:
  1. Portal persiste a solicitação localmente com um `idPortal`
  2. API ACL cria a requisição no RM usando o **RH Central** como requisitante técnico
  3. API ACL abre o atendimento/fluxo no "Minhas Aprovações"
  4. Portal registra os identificadores retornados pelo RM (`idRmRequisicao`, `idRmAtendimento`)
  5. Usuário é redirecionado para `/portal/solicitacoes/:id`
- Em caso de falha na integração: exibe mensagem de erro clara, mantém solicitação com status "Falha ao enviar" e permite reenvio

---

#### ⚙️ Tela: Administração / Parâmetros

**Rota:** `/admin/parametros`<br />**Perfil:** Somente **Administradores**

**Objetivo:** Permitir configuração do **RH Central** (requisitante técnico) e gerenciamento de permissões, sem necessidade de alteração de código.

**Elementos:**

- Lista de **usuários RH Central** configurados (por empresa/contexto)
- Botões para adicionar/editar/remover entradas
- _(Fase 2: gerenciamento de páginas Markdown)_
